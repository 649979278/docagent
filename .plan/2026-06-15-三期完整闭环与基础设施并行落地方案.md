# 三期完整闭环与基础设施并行落地方案

## 摘要
- 本期按“业务闭环 + 基础设施一起做”执行，但顺序固定：先恢复工程可运行性，再完成存储与运行时重构，再收口 RAG 和 Worker 真链路，最后补双层门禁测试与 CI。
- 目标不是“文件都补齐”，而是让计划审批、执行、知识库、检索、恢复、中断、桌面构建、真实 E2E 全部跑通，且测试能证明没有业务卡点。
- 本期交付后的验收标准是：`pnpm install`、`pnpm doctor:env`、`pnpm test:unit`、`pnpm test:e2e`、`pnpm -r build`、`pnpm --filter @workagent/desktop build`、`pnpm verify` 在本地可跑通；CI 采用双层门禁，普通 CI 跑无模型链路，真实 Ollama + LanceDB 链路作为本地全量门禁和可选 self-hosted job。

## 关键接口与架构调整
- `packages/store/src/database.ts`
  `initDatabase()` 改为同步返回，底层从 `sql.js` 切到 `better-sqlite3`。现有 `Database`/`Statement` 包装层保留，但内部实现替换为 native driver，移除 `save()` 和 `getSqlJsDb()`。
- `packages/rag/src/components.ts`
  `ChunkMetadataStore` 去掉 `documentId` 回流字段；RAG 包只处理 `chunkId/sourceFile/content`，文档主键绑定放到 desktop 应用服务层。
- `packages/rag/src/engine.ts`
  `indexDocument()` 只负责 chunk + embedding + vector upsert，不再直接承担 document/chunks SQL 事务语义；新增由调用方提供的 metadata callback 或 service 编排。
- `apps/desktop/electron/services/knowledge-service.ts`
  新增知识库应用服务，统一承接 `knowledge-add`、`knowledge-remove`、`knowledge-remove-batch`、worker index、tool index，保证 document/chunks/FTS/vector/workspace 全部一致。
- `apps/desktop/electron/services/plan-service.ts`
  新增计划应用服务，统一承接 `plan-mode`、`plan-approve`、active plan 恢复、执行启动、输出回写，消除 IPC、worker、runtime 三处重复状态写入。
- `packages/shared/src/tokens.ts`
  新增统一 token 计数入口，`agent-core`、`rag` 共用；仅 main/worker/store 路径引用，不暴露到 renderer。
- `packages/shared/src/logger.ts`
  新增统一 pino 日志工厂，替换 Electron main、runtime、RAG、store 中的散落 `console.*`。
- `apps/desktop/electron/runtime-factory.ts`
  runtime 装配从“工具拼装”升级为“provider + store + knowledge service + plan service + worker parity”的单一入口；测试 bundle 与生产 bundle 共享装配路径，不再走特判逻辑。

## 实施步骤
### 1. 阻塞项与环境修复
- 执行 `pnpm install`，修复缺依赖导致的 `vitest` 不可用问题。
- 修正 `tests/e2e/rag-quality-regression.test.ts` 的语法/缩进错误，并顺带跑根级 `vitest` 语法检查，避免后续长链路才暴露解析失败。
- 在根 `package.json` 增加 `pnpm.overrides["@noble/hashes"]`，同步检查 desktop build 和 LanceDB 依赖链是否受影响。
- 修补 `scripts/run-vitest.js` 和根脚本，确保 e2e runner 对 Windows + pnpm workspace 解析稳定；若当前脚本仅依赖 `node_modules/.pnpm`，要补清晰的缺依赖报错。
- 扩展 `scripts/doctor-env.js`：除 Node/pnpm/Ollama/模型外，再检查 `better-sqlite3` 可加载、LanceDB 依赖可初始化、workspace 依赖已安装。

### 2. 存储层迁移到 better-sqlite3
- 替换 `packages/store/src/database.ts` 的底层实现，保留现有包装 API 以减少上层改动面积，但彻底移除 `sql.js` 的导出/落盘模式。
- 更新 `packages/store/src/migrations/*.ts`，尤其是 `004_chunks_fts.ts` 的 `PRAGMA table_info`、事务和返回行格式适配。
- 全仓库移除 `db.save()` 调用，改为 native 自动持久化语义；重点修改 `apps/desktop/electron/*`、`packages/agent-core`、`packages/store` 各 repository。
- 调整 Electron worker、main 进程、测试环境的 native 模块加载路径，并同步检查 `electron-builder.yml` 是否需要补 `better-sqlite3` 打包配置。
- 补充 `packages/store` 的 CRUD、事务、迁移、FTS、并发读写测试，确保不因底层替换改变业务语义。

### 3. Plan 执行链路彻底闭环
- 在 `plan-service` 中定义唯一状态流：`chat -> plan(draft) -> approved -> execute(executing) -> completed/cancelled`。
- 修改 `apps/desktop/electron/ipc/chat-ipc.ts`、`workers/agent-worker.ts`、`plan-persistence.ts`，让 `plan-approve` 以 `sessionId + planId` 为主键落盘，不再依赖 worker 内存里的 `currentSessionId`。
- `packages/agent-core/src/runtime.ts` 保留“下一次 execute runTurn 检测 approved 后 startExecution”的产品语义，但执行前必须通过 service 从 DB/session 恢复 active plan，避免 runtime/controller 状态与 DB 脱节。
- 前端审批条继续在 `ConversationPane.tsx`，但其行为改为只发命令，不自己承担状态真相；UI 状态只由事件流和 session-resume 恢复。
- 输出闭环要求：`draft_ready/doc_ready` 必须回写 plan 记录、session activePlan、resume snapshot；中断和拒绝计划必须清除 activePlanId。

### 4. 知识库与 RAG 真闭环
- 新增 `knowledge-service` 负责完整编排：幂等检查、extract、chunk、embedding、vector upsert、chunks SQL、FTS 同步、document 状态、workspace 关联、重建与删除。
- `ChunkMetadataStore` 从 RAG 包剥离 document 维度；desktop adapter/service 负责把 `documentId` 和 chunk metadata 写入 `chunks`。
- `knowledge-add` IPC、`KnowledgeAddTool`、index-worker 统一走 service，不再一套走 job mock、一套走真实索引。
- `knowledge-remove` 和 `knowledge-remove-batch` 必须同时清理 `documents/chunks/chunks_fts/vector index`，但不删除原始文件；`reconcileDeletedKnowledge` 也走同一 service。
- `BM25Search`、`004_chunks_fts`、`RAGEngine.removeDocument()` 的语义统一成“sourceFile 删除 vector、documentId 删除 SQL metadata”，由 service 负责映射，避免分层错乱。
- `resumeSession` 增强为 transcript + DB 混合恢复：若 transcript 缺 `plan_generated/approved` 事件，仍可从 `plans/sessions` 恢复 activePlan 快照。

### 5. Token、日志、真实 RAG 组件
- 用 `js-tiktoken` 替换字符估算，统一替换 `packages/agent-core/src/context/budget.ts`、`compact.ts`、`retrieval-pipeline.ts` 等所有预算判断入口，避免不同模块估算口径不一致。
- 新增共享 logger，并在 store、runtime、ipc、runtime-factory、worker、rag 里替换 `console.*`；日志字段至少包含 `component/sessionId/runId/planId/documentId`。
- `BGEReranker` 以 Ollama `/api/rerank` 为主实现，保留 chat-based rerank 作为运行时降级，不再把 `PassThroughReranker` 视为默认主链。
- `OllamaQueryRewriter` 改为真实调用模型输出 JSON 数组或关键词串，失败再降级到规则重写；默认链优先使用 Ollama 版本，只在模型不可用时回退。
- `runtime-factory` 的 `retrievalDiagnostics` 必须反映当前真实主链和实时 fallback 状态，前端与测试都据此断言。

### 6. Worker 默认启用与 LanceDB 真跑
- 把 worker 模式从“尽量启用”改为“默认启用，握手失败才回退”，并把失败原因显式打日志和 diagnostics。
- `runtime.worker-parity.test.ts` 扩展为主进程 direct 与 worker 模式在 `run_status / plan events / abort / output events` 上完全对齐。
- `createVectorStore()` 在生产和真实 E2E 都优先使用 LanceDB；只有显式 mock/unit 测试才允许 `MemoryVectorStore`。
- 为真实 RAG E2E 增加专用临时目录和清理逻辑，确保 LanceDB 数据、better-sqlite3 DB、transcript artifacts 都可重复执行。
- 补充高质量 fixtures：政策文号、标准规范、会议纪要、公文草稿、报告正文，至少覆盖 dense 命中、sparse 命中、remove 后失效、restart 后恢复四类检索场景。

## 测试与门禁
- 单元测试：
  覆盖 better-sqlite3 迁移、token 计数、logger 工厂、plan service、knowledge service、reranker、query rewriter。
- 集成/E2E：
  1. `前端审批 -> IPC -> DB execute -> 下一轮 runTurn -> executing -> draft/doc 输出`
  2. `knowledge-add -> hybrid search -> remove -> restart -> search miss`
  3. `worker chat -> abort exactly once -> run_status 对齐 -> resume snapshot 可恢复`
  4. `LanceDB + FTS5 + better-sqlite3` 真实链路
  5. `plan reject / cancel / completed` 三种终态恢复
- CI 双层门禁：
  普通 CI 运行 `pnpm install`, `pnpm test:unit`, `pnpm test:e2e` 的无模型集合, `pnpm -r build`, `pnpm --filter @workagent/desktop build`。
  真实模型门禁通过环境变量分组，例如 `E2E_REAL_OLLAMA=1` 跑 Ollama + LanceDB + Worker 真链路；默认不阻塞公共 CI，但阻塞本地 `pnpm verify` 和可选 self-hosted job。
- `pnpm verify` 重定义为本地全量门禁：
  先 `doctor:env`，再普通测试，再真实 RAG/Worker E2E，再全量构建。

## 假设与默认决策
- 本期接受较大改动，包含 `better-sqlite3`、`js-tiktoken`、`pino`、Worker 默认启用、LanceDB 真实 E2E、CI/规范配置。
- 若 `better-sqlite3` 在 Electron Worker 中验证失败，不回退整个方案，而是保留 `better-sqlite3` 作为主实现，同时让 Worker 通过独立数据库访问适配层修正加载方式；只有连续验证失败才把 fallback 记录为下一轮决策。
- 本期会补测试，不按“小改动不补单测”处理，因为这是跨模块闭环修复。

## 实际桌面应用测试复盘
- 真实启动方式：使用 Electron 可执行文件启动 `apps/desktop` 已构建应用，通过 UI 新建会话并发送真实问题，模型使用本地 `qwen3.5:9b`。
- 暴露问题 1：`better-sqlite3` 先按 Node ABI 编译，真实 Electron 运行时报 `NODE_MODULE_VERSION 137/146` 不匹配，导致 `session-list/workspace-list` IPC 失败。
- 修复问题 1：新增 `@electron/rebuild`、`desktop:prepare-native`、`scripts/check-electron-native.js`，并在 `electron-builder.yml` 中解包 `better-sqlite3` native 文件；真实桌面启动前显式执行 Electron ABI rebuild，测试完成后用 `native:restore-node` 恢复 Node ABI。
- 暴露问题 2：实际模型可用并能回复，但 UI 仍显示“未配置/离线”。
- 修复问题 2：新增 `apps/desktop/src/utils/model-status.ts`，App 启动时主动调用 `getModelsStatus()` 并订阅 `ollama-status`，兼容原生 provider 返回结构和旧 preload 契约。
- 暴露问题 3：恢复历史会话时消息时间显示 `Invalid Date`，根因是 `session-messages` IPC 返回 store 层 `createdAt`，而 preload/renderer 契约要求 `timestamp`。
- 修复问题 3：在 `apps/desktop/electron/ipc-handlers.ts` 增加 `toSessionMessageDto()`，在 IPC 边界统一把 `createdAt` 映射为 `timestamp`。
- 暴露问题 4：第二轮真实对话已经完成并落库，但 UI 仍显示“生成中/停止”，根因是前端只在 `done/error` 关闭 loading，未把 `run_status: completed/aborted/failed` 作为终态闭环。
- 修复问题 4：`apps/desktop/src/hooks/useAgentEvents.ts` 增加 `finishCurrentRun()`，让 `run_status` 终态、`done`、`error` 统一关闭流式状态。
- 实测结论：第三轮真实 UI 对话完成后，页面显示 `qwen3.5:9b/在线`，停止按钮消失，无 `生成中`，无 `Invalid Date`，消息已通过 `sessionMessages()` 持久化恢复。
- 补充测试：新增 desktop 模型状态映射测试，更新 preload 类型契约测试，将 `@workagent/desktop` 纳入 `pnpm test:unit`。
