# WorkAgent 第三期计划文档

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在不推翻一期、二期已验证资产的前提下，完成 WorkAgent 第三期真正可执行、可测试、可交付的落地计划，先闭合一期未完成断层，再把 Runtime、RAG、Desktop 工作台和发布验证补到接近 Claude Code 与顶尖开源知识库 Agent 的工程水准。

**Architecture:** 第三期不做整仓重写，采用“保留骨架 + 替换关键路径 + 延后非核心高级能力”的策略。核心动作是先闭合一期遗漏的运行路径，再对 `ipc-handlers.ts`、`agent-worker.ts`、`retrieval-pipeline.ts` 做替换式重构，并把恢复、索引、检索、桌面状态和测试链路做成闭环。

**Tech Stack:** Electron、React 19、TypeScript、pnpm workspace、sql.js、LanceDB、Ollama、Vitest、Playwright、electron-builder、Node 24

## 一、总判断

### 1.1 是否需要完全重写

结论：

- **不需要完全重写整个项目**
- **需要替换式重构 3 条关键路径**
- **需要增量增强 4 个基础模块**
- **需要把部分高级能力明确降级到第三期后半段或第四期**

原因：

- 当前项目的骨架方向没有错。
- 错误主要不在“选型”，而在“真实运行路径没有闭环”和“同一能力存在两套不一致实现”。
- 一期、二期已沉淀的 `QueryLoopState`、compact、Plan 控制器、ToolExecutor、IngestPipeline、KnowledgeIndex、SQLite/LanceDB 边界都值得保留。
- 如果全量重写，会损失现有类型、存储、工具、安全边界和 UI 状态结构，收益远低于替换式重构。

### 1.2 第三期必须先纠正的几个方案错误

以下是你新版方案里我建议直接修正的点：

1. **不能把三期前半段建立在“一期未闭合”的代码上**
   - 这一点你已经识别出来了，正确。
   - 所以第三期正式起点必须是“前置阶段 0：一期闭环修复”，不是直接进入 BM25 / rerank。

2. **`retrieval-pipeline.ts` 不能继续按单文件叠功能**
   - 这一块不是简单新增几个阶段。
   - 应该先拆成阶段式实现和统一上下文对象，否则后面 rewrite / fusion / rerank / grade 叠加后会快速失控。

3. **BGE reranker 不能在计划里假定一定用 `/api/chat` 就能稳定得到可信评分**
   - 这在实现上风险较大。
   - 正确做法是先定义 `Reranker` 抽象接口，Ollama 实现是可选能力；若模型不可用或效果不稳定，必须自动降级为 pass-through，并且通过质量测试决定是否默认启用。

4. **Windows 安装包验证不能伪装成“完全自动化测试”**
   - 在 Mac 上可以做“静态打包验证”。
   - 但真正的 Windows 安装、快捷方式、路径、Ollama 探测、卸载行为，仍需要 Windows 真机或虚机验收。

5. **知识图谱 Lite 不应进入第三期前半段**
   - 有价值，但不是当下成败关键。
   - 当前更关键的是：Worker/Direct 一致性、Plan 闭环、Knowledge Lifecycle、Hybrid Retrieval、Recovery。

6. **自动批准类权限功能不应早于权限闭环本身**
   - 先把 Hook、权限状态、用户确认路径做实。
   - Transcript classifier 放到第三期后半段更合理。

## 二、保留 / 重构 / 延后三分法

### 2.1 保留

这些模块不应推倒：

- `packages/agent-core/src/query-state.ts`
- `packages/agent-core/src/plan-controller.ts`
- `packages/tools/src/executor.ts`
- `packages/ingest/src/pipeline.ts`
- `packages/rag/src/knowledge-index.ts`
- `packages/store/*`

保留原因：

- 边界已基本成立。
- 类型关系清晰。
- 继续增强成本低于重写。

### 2.2 替换式重构

这些模块应按新边界重组：

- `apps/desktop/electron/ipc-handlers.ts`
  - 当前是总控 God object。
  - 第三期应拆成：
    - `runtime-factory.ts`
    - `chat-ipc.ts`
    - `knowledge-ipc.ts`
    - `settings-ipc.ts`
    - `worker-runtime-bridge.ts`

- `apps/desktop/electron/workers/agent-worker.ts`
  - 现在是简化 Runtime。
  - 必须改成和 direct 模式共用装配路径。

- `packages/rag/src/retrieval-pipeline.ts`
  - 从“单文件串流程”改成“阶段化管道”。
  - 需要按：
    - normalize
    - rewrite
    - dense
    - sparse
    - fusion
    - rerank
    - grade
    - truncate
    - pack
    做成可组合阶段。

### 2.3 增量演进

- `packages/agent-core/src/runtime.ts`
  - 保留主循环。
  - 拆出 transcript、drain collapse、streaming tools、max_output_tokens recovery。

- `apps/desktop/src/components/*`
  - 不重写 UI 壳。
  - 增加 tool result viewer、plan visualizer、knowledge status、run status。

### 2.4 延后

这些能力不进入第三期前半段：

- `knowledge-graph lite`
- `transcript classifier 自动批准`
- `复杂实体关系检索`
- `过多新工具`
- `过度花哨的 UI 效果`
